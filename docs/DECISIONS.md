## 2026-05-09 — 호스트 계정 운영 방식 (옵션 A) + 인증 모델 결정 보류

- **결정**: 필드 테스트 단계의 호스트 계정 운영 방식을 옵션 A(Firebase 콘솔 수동 생성)로 확정. 인증 모델 자체의 결정(계정 모델 vs 이벤트-바운드)은 필드 테스트 첫 회차 결과를 본 후 재결정으로 보류.
- **이벤트-바운드 모델 채택 결정도 보류 처리**: 영향 범위가 갈래 1A~1G 7개 갈래 규모로 큰 작업이며, 필드 테스트 첫 회차 결과 보고 모델 자체에 대한 새 발견이 나올 수 있음. 모델 작업을 시작하기 전 진짜 사용 패턴 데이터 확보가 정합.
- **옵션 A 운영 절차**:
  - 필드 테스트 호스트마다 운영자가 Firebase 콘솔에서 사용자 추가 (Authentication → Users → Add user)
  - 이메일 + 임시 비번을 카톡/문자로 호스트에게 전달
  - 호스트는 /host에서 로그인 (현재 코드 그대로 동작)
  - 사용자가 비번 분실 시 기존 비번 재설정 모달로 자가 복구
  - 필드 테스트 종료 후 콘솔에서 계정 일괄 삭제 + 해당 hostId의 events 정리
- **이유**:
  - 코드 변경 0. 모델 결정과 완전 독립.
  - 운영자가 자기 계정 만들 때 한 번 해본 흐름 재사용.
  - 필드 테스트 N명 규모(10~30명 × 2~3회)에서 호스트는 회당 1~2명, 총 5~10명 콘솔 작업 — 1인당 30초로 부담 작음.
  - 모델 결정을 첫 회차 후로 미뤄서 작업 두 번 막음.
- **트레이드오프**: 운영자 수동 부담 + 비번을 운영자가 알게 됨. 1차 단계 임시 운영이라 수용 가능.
- **재결정 시점**: 필드 테스트 첫 회차 종료 후 인터뷰에서 모델 결정.
- **대안**: 가입 페이지 신설 (이벤트-바운드 전환 포함) — 7개 갈래 규모, 필드 테스트 데이터 없이 진행하는 비용 큼 → 보류.

## 2026-05-09 — 호스트 인증 결정 폐기 + 이벤트-바운드 모델 채택

- **결정**: 같은 날(2026-05-09) 등록된 "호스트 인증: 가입/로그인 페이지 분리" 결정을 폐기. 대신 이벤트-바운드 모델로 전환.
- **이벤트-바운드 모델**:
  - 가입 개념 없음. hosts 컬렉션 만들지 않음. 영속 호스트 계정 없음.
  - 이벤트 생성 폼에 이메일/이름/휴대폰번호/약관동의 입력. 비밀번호 없음.
  - 서버가 이벤트 생성 시 hostToken 발급. `events/{eventId}.hostToken` 필드에 저장.
  - 호스트에게 이메일+SMS로 관리 링크 발송 (예: `/dashboard/events/{eventId}?hostToken=...`).
  - 호스트는 그 링크로 이벤트 관리 진입. 토큰 분실 시 "이메일로 관리 링크 다시 받기" 흐름으로 복구.
  - 이벤트 + 호스트 정보가 한 묶음. 이벤트 삭제 = 호스트 정보 삭제. 데이터 보유 기간이 이벤트 수명과 동일.
- **이유**:
  - 데이터 최소 수집 원칙 정합. 이벤트 1회를 위해 영속 계정 만들 필요 없음.
  - 1순위 시장(학교 졸업식) = 미성년자. 미성년자 데이터의 영속 보유는 자산이 아니라 법적 부채.
  - 1년 1~2회 사용 빈도에서 월 단위 재방문 SaaS형 마케팅 모델은 한계 효용 작음. 영상 공유 자체가 더 큰 마케팅 채널.
  - 정찰에서 `organizerPhone`이 이미 이벤트별 저장 구조 확인 — 코드는 사실상 이벤트-바운드를 부분적으로 이미 구현 중. 계정 모델은 이걸 뒤집는 변경.
  - 비밀번호 제거 이유: 이벤트당 1회 사용에서 비번은 토큰 역할. 그러면 그냥 토큰을 쓰는 게 정직.
  - 탈퇴 모듈 불필요. 운영 부담 감소.
- **마케팅 데이터 자산화 의제 (보류)**: 향후 BM이 B2C 1회성 외 방향으로 이동 시 재평가. 마케팅 동의는 별도 약관 항목으로 받을 수 있으므로 이벤트-바운드 모델에서도 동의 기반 마케팅 가능.
- **Firebase Auth 처리**: 끊지 않고 그대로 둠. 사용 안 함. 향후 되돌리기 옵션 보존.
- **대안**: 계정 모델(가입/로그인 분리, hosts 컬렉션 신설) — 위 이유들로 기각.

## 2026-05-09 — 재렌더 정책 (D1): 횟수 무제한, 경고 모달, 실서비스 시 사전 결제

- **결정**: 재렌더 횟수 무제한. 사용자가 [영상 다시 만들기] 클릭 시 경고 모달 노출. 실서비스 단계에서는 사전 결제 게이트 추가. 필드 테스트 단계는 결제 없이 경고 모달만.
- **추가 결정**: 재렌더 버튼을 `status === "done"` 상태에도 노출 (현재는 `"closed"`에서만 노출). 클릭 시 클립 토글 모달로 제외/포함을 다시 선택할 수 있게 함.
- **이유**:
  - 운영자 사용 패턴 인식상 "완성 영상 보고 클립 다시 선택해서 재렌더"가 흔한 케이스. 빈도 높음 + 복구 경로 부재 = 우선순위 격상.
  - 종량제 과금 모델에서 재렌더 비용은 사용자 부담. 횟수 제한은 사용자 가치를 깎고 운영자 부담은 안 줄임.
- **보류 의제** (실서비스 진입 시점에 결정): 사전 결제 시점이 종량제와 어떻게 결합되는지. 결제 모델 설계 의제로 별도 트랙.
- **대안**:
  - 재렌더 N회 제한: 사용자 가치 손실 + 종량제 모델과 부정합 → 기각.
  - 클립 토글 모달 없이 사전 토글 후 재렌더만: 발견율 낮음, 사용자 학습 비용 → 기각.

## 2026-05-09 — 호스트 인증: 가입/로그인 페이지 분리, 1차안에서 인증 절차 최소

- **결정**: 호스트 인증을 가입 페이지 + 로그인 페이지 분리 구조로 정비. 가입 입력 필드: 이메일, 비밀번호, 휴대폰번호, 이용약관 동의.
- **이유**:
  - 정찰에서 가입 흐름이 코드에 없음을 확인 (signup 함수 없음, 운영자 계정은 Firebase 콘솔 직접 생성). 신규 이메일로 로그인 시 "비번 또는 이메일이 올바르지 않음" 에러로 막다른 길 → 두 번째 호스트 검증 자체가 불가능.
  - 1차안에서 이메일 인증·SMS 인증·자체 비번 찾기는 제외. Firebase Auth 기본 `sendPasswordResetEmail`은 그대로 유지.
- **대안**:
  - 신규/재방문 분기 (한 화면에서 이메일 입력 후 분기): 이메일 enumeration 보안 이슈 + 분기 API·UI 상태 관리 추가 비용 → 기각.
  - 가입 시 이메일·SMS 인증 즉시 도입: 발송 비용 + 도달성 의제(known-issues 네이버 메일) 미해결 상태에서 마찰만 늘림 → 1차안 제외.

## 2026-05-09 — 시장 정의 보강: 단체사진 영상 버전, 자동 편집 필수, 제작자 무관

- **결정**: congre는 "단체사진으로 남는 모든 행사의 영상 버전"이다. 결혼식·졸업식·사내행사·MT 등 행사장에서 축하 인사·소감·현장 분위기를 누군가가 담아 주최자에게 보내고, 주최자가 감사 인사·마무리 멘트를 더해 실시간 완성본을 받아 참가자·SNS로 공유하는 도구.
- **이유**:
  - 1순위 시장(학교 졸업식, DECISIONS 2026-05-06)의 "1인칭 자기 회고" 패턴은 시장 안에서의 활용 형태이지 서비스의 본질이 아님. 결혼식 하객 축하·사내행사 동료 코멘트 등은 타인 시점이 자연스러움.
  - 자동 편집은 비용 우위가 아니라 현장성 보존을 위한 필수 조건. 사람 편집자(WeddingMix류)는 이 가치와 양립 불가.
  - 제작자 = 폰 가진 누구나 (행사 주최자, 학생, 사원 등). UI는 특정 역할 가정 없이 흘러야 함.
  - 영상 가치 강화 방향(DECISIONS 2026-05-05의 6항목) 중 인물 보정·배경 정리는 "실제 얼굴 + 현장성" 가치와 충돌. 강화 방향은 행사 컨텍스트 표시(가상 포토월, 행사 타이틀) 쪽으로 좁힘.
- **대안**: 1인칭 회고 전용 도구로 좁히기 — 결혼식·사내행사 시장 진입 차단으로 기각.

## 2026-05-08 — 갈래 패턴: 정찰 → 코드 커밋 → 문서 커밋

- **결정**: 큰 기능은 갈래로 분리. 각 갈래: 정찰(코드 변경 없음) → 코드 atomic 커밋 → 다음 갈래 정찰... → 완료 후 문서 갱신 atomic 커밋 1개.
- **이유**: 정찰이 실행 범위를 결정하므로 선행 필요. 문서는 완료된 사실을 기록하므로 코드 후행. 중간 문서 갱신은 미완성 상태를 기록하게 되어 혼란 유발.
- **적용**: 인트로/아웃트로 작업이 초안 — 갈래 1(폰트) → 갈래 2(UI/API) → 갈래 3(Shotstack 연결 + 디버그 + 미세 수정) → 갈래 4(문서) 순서.

## 2026-05-08 — Shotstack rich-text asset 채택, timeline fonts 주입, length 3초 고정

## 2026-05-08 — 클립 제외 기능: excludedAt Timestamp 필드, JS 필터, 양방향 토글

- **결정**: 클립 제외 상태를 `clips.excludedAt: Timestamp | null` 필드로 관리. `render/start`에서 Firestore 조회 후 JS 필터(`!d.data().excludedAt`)로 적용. 주최자 대시보드 클립 행에 양방향 토글 버튼 (Eye/EyeOff 아이콘). 낙관적 업데이트 후 API 실패 시 롤백.
- **이유**:
  - Timestamp는 boolean 대비 언제 제외했는지 디버깅 정보를 보존. 미존재 필드(기존 클립)와 null(복원된 클립)을 동일하게 처리 가능.
  - JS 필터: Firestore `where("excludedAt", "==", null)` 사용 시 `excludedAt` 필드 미존재 문서가 쿼리에서 누락됨 (Firestore 동작 특성). JS 필터는 이 사고를 원천 차단 + composite index 불필요.
  - 양방향 토글: 제외 후 실수 복구 가능. 주최자 UX 안전.
- **대안 검토**:
  - `excluded: boolean` 필드: 시각 정보 없음. boolean → timestamp 마이그레이션은 정보가 없어 비용이 큰 반면 timestamp → boolean은 정보 줄이는 방향이라 거의 안 일어남. 비대칭 비용 → 기각.
  - Firestore where 절 필터: 미존재 필드 사고 위험 + composite index 필요 → 기각.
  - 단방향 삭제(Firestore doc delete): 실수 복구 불가 → 기각.
- **전체 제외 가드**: JS 필터 후 0개면 `400 NO_CLIPS_AFTER_EXCLUSION`. 기존 `NO_CLIPS`(클립 자체 0개)와 에러 코드 구분.

## 2026-05-07 — 서비스 모델 정의: 비공식 기념품, 즉시 삭제 데이터 흐름

- **결정**: congre는 학교·기업 등 주최 측이 공식적으로 발주하는 행사 솔루션이 아니라, 참가자들끼리 사적으로 기념하는 도구로 정의한다. 데이터는 인터넷 망에 영구 보존하지 않고 단기 처리 후 삭제한다.

  - 완성 영상: 주최자에게만 전송 → 주최자가 참가자에게 공유 (공유 인프라는 congre가 제공) → 며칠 후 삭제
  - 주최자 공유 이후의 배포 책임은 congre 영역 밖
- **함의 — 보안·법무 판단의 공통 전제**:

  - 보안 작업 우선순위는 "유출"보다 "기능 신뢰성"(렌더 성공률, 알림 도달성) 쪽에 비중을 둔다.
- **적용**:
  - 향후 모든 보안·법무·기능 결정은 이 데이터 흐름을 전제로 평가한다.
  - 이 정의가 바뀌면(예: 영상 영구 보존 옵션 도입, 학교 공식 발주 진입) 본 항목을 업데이트하고 보안 우선순위를 재평가한다.

## 2026-05-07 — Cron 이전: GitHub Actions → Vercel Cron, Pro 업그레이드

- **결정**: GitHub Actions schedule cron을 폐기하고 Vercel Cron으로 이전. Vercel Pro 플랜으로 업그레이드 ($20/월).

- **대안 검토**:
  - Shotstack webhook 도입 (cron 유지): 시간 카운트(render_delayed)는 webhook으로 못 함. 보완재이지 대체재 아님.
  - 외부 cron 서비스 (cron-job.org 등): 운영 책임 분산, 1인 비개발자에게 트레이드오프 나쁨.

- **인증**: Vercel Cron이 호출 시 `Authorization: Bearer ${CRON_SECRET}` 헤더 자동 주입. 라우트 코드 수정 불필요.
- **정리**: GitHub Actions 워크플로 yml 2개 삭제. GitHub Secrets (APP_URL, CRON_SECRET)는 사용자가 GitHub UI에서 별도 삭제.

## 2026-05-06 — 클립 길이 정책: 인당 10초 이내

- **결정**: 사용자 1명이 올리는 클립 길이는 **10초 이내**.
- **이유 (3가지가 동시에 같은 답)**:

## 2026-05-04 — render_delayed 시나리오 정책 확정

### 결정
render_delayed 시나리오를 단일 +10분 알림에서 다단계 시간축으로 재설계한다.
T=0(마감 버튼) → T+E(예상 완료) → T+E+30분(50% 환불 확정) → T+24시간(100% 환불 확정).

### 핵심 정책
1. **T=0 기준점**: 주최자가 "마감하기" 버튼을 누르는 시점 (기존 deadlineAt 활용)
2. **E 산정 공식 (1차안)**: `E = max(15분, 클립수 × 30초 + 10분)`. 필드 테스트 후 조정.
3. **클립 수 소스**: render/start 시점에 클라이언트가 전달하는 s3Keys.length 사용.
   events.clipCount는 참가자(비로그인) 업로드 실패 가능성 때문에 신뢰 불가.
4. **환불 확정의 비가역성**: T+E+30분을 1초라도 넘으면 50% 확정. 이후 완료되어도
   번복하지 않음. 24시간 넘으면 100% 확정 + 영상은 끝까지 완성하여 전달.
5. **환불 처리는 알림만 자동, 실행은 수동**: 카카오톡 채널 @congre로 CS 응대 시 처리.
6. **품질 클레임 미접수**: 별도 결제 동의서에 명시 (별도 작업).
7. **고객 알림 횟수**: 시나리오당 1회 원칙. 중간 리마인더 없음.
8. **사내 담당자 알림**: SMS 이중 발송 (슬랙 도입 시 이전).
