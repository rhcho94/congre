## 오늘 시행착오 — 패턴별 정리

이 섹션이 다음 세션의 핵심 가치. 같은 사고 반복 안 하기 위해.

### 사고 1: webhook 잠금이 prod 핵심 기능 망가뜨림

관련 커밋: 2d28780, a62fb4e → revert

**원인:** `/api/render/complete`를 dead 엔드포인트로 판단하고 401 잠금. 그러나 정찰이
"createRender의 callback URL"만 봤지 "클라이언트가 호출하는가"는 grep 안 함.
실제로는 `dashboard/events/[eventId]/page.tsx`의 폴링이 이 엔드포인트를 호출하고 있었음.

**교훈:** "X를 누가 호출하는가" 전수 grep을 모든 보안 작업의 첫 단계로.
dead 판정은 호출자 0건 확인 후에만.

---

### 사고 2: 알림 fire-and-forget → 실제 발송 안 됨

관련 커밋: cbfb973

**원인:** `notifyXxx({...}).catch()` 패턴은 `await` 없어서 Vercel serverless가 응답 반환 시
컨테이너 종료 → Promise 미실행. 코드 검토 시 catch만 있으면 충분하다고 잘못 판단.

**표준 답:** Vercel + Next.js App Router에선 `await` 또는 `waitUntil` 필수.
fire-and-forget은 100% 사고.

**교훈:** 외부 API 호출이나 알림 발송 코드 보면 즉시 호출 패턴 점검 (await/waitUntil/
fire-and-forget). 이건 Vercel serverless의 표준 함정.

---
