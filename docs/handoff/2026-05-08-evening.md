## 1순위 — 네이버 메일 도달성 점검 결과 (보류 판정)

운영자 직접 콘솔 작업 + 외부 검증 + 1통 실측으로 4단계 점검:

- Resend 도메인 verified ✅, DKIM/SPF/DMARC 모두 등록 ✅
- MXToolbox 외부 검증으로 DKIM public key Resend 콘솔과 동일성 확인 ✅
- DMARC `p=none` 정상 (도입 초기 권장 정책)
- 네이버 메일 1통 실측: 받은편지함 도달, 경고 배너 없음 ✅

**잠재 리스크 잔존:** From 주소 `noreply@congre.kr` (루트)인데 SPF는 `send.congre.kr` (서브)에 등록됨. SPF alignment 측면 mismatch이지만 DKIM alignment로 DMARC 통과 중인 것으로 추정. 신규 도메인 평판이 쌓이지 않은 상태에서 발송량 증가 시 도달성 저하 가능성.

`docs/known-issues.md`에 1차 점검 포인트로 메모 등록. 점검 트리거: 사용자 신고 발생 시.

### 갈래 3 사고 — width/height 미지원 필드

**경위:** 갈래 3-B 첫 시도에서 마감 시 Shotstack 400 거절. 클라이언트 alert 메시지가 200자에서 잘려 진단 지연. `shotstack.ts:106` `text.slice(0, 200)` 처리가 본문 자른 원인. 디버그 로그 추가 후 errors 배열 두 항목에서 정확히 width 필드만 짚음.

**근본 원인:** rich-text asset에 width/height를 spec 단계에서 박아 보냄. CC가 작업 직전 가이드 재fetch로 align/alignment, "center"/"middle" 차이는 잡았지만 width/height까지 1:1 대조하지 않음. 정찰 단계에서 가이드 텍스트 설명만 봤고 JSON 예제 전체를 복사하지 않은 게 근본.

**자체 학습:** DECISIONS 2026-05-08에 "외부 API 통합 작업의 spec 검증 패턴" 등록. 향후 외부 API JSON spec은 가이드 예제 복사 → 우리 사양 변경 → 1:1 대조 절차로 진행.
